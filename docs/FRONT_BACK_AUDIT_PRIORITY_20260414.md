# 前后端代码审计记录（2026-04-14）

## P0（必须优先处理）

### 1) 管理类接口缺少鉴权与授权边界
- 证据:
  - 配置写入接口可直接调用: api/routes.py:85
  - 日志删除接口可直接调用: api/logs.py:87
  - 模型训练/导入接口可直接调用: api/model.py:100, api/model.py:200
  - 补采启动/重试接口可直接调用: api/backfill.py:146, api/backfill.py:177
- 风险:
  - 未授权用户可修改系统配置、清理日志、触发高负载任务，造成数据与服务风险。
- 建议:
  - 引入统一鉴权中间件（至少管理员密钥或登录会话）。
  - 对写操作接口增加角色校验（admin/operator/viewer）。
  - 对高风险接口增加审计日志（操作者、来源 IP、参数、结果）。

### 2) 模型导入存在路径与内容安全风险（可与反序列化风险联动）
- 证据:
  - 导入时直接使用上传文件名拼接路径: api/model.py:225
  - 直接保存上传文件: api/model.py:226
  - 系统存在 pickle.load 读取模型/数据链路: api/reviews.py:67, scheduler.py:470
- 风险:
  - 文件名若包含路径片段可能造成路径穿越写入风险。
  - 未校验内容类型，恶意 pickle 文件被后续加载时可触发远程代码执行。
- 建议:
  - 对上传文件名使用安全文件名规范化（如 secure_filename）并禁止路径分隔符。
  - 限制扩展名与 MIME，加入文件大小上限与签名校验。
  - 关键模型文件改为受信任签名校验后才能加载；优先替换为更安全序列化格式。

### 3) 默认敏感配置不安全（生产误配置风险）
- 证据:
  - SECRET_KEY 有弱默认值: config.py:38
  - TUSHARE_TOKEN 提供了非空默认值: config.py:97
  - 服务默认绑定 0.0.0.0: config.py:40
- 风险:
  - 生产环境误用默认值会降低整体安全性。
- 建议:
  - 启动时强校验：生产环境缺少关键环境变量则拒绝启动。
  - 将示例 token 彻底移到 .env.example，代码默认值置空。

## P1（应尽快处理）

### 4) 缺少 CSRF 保护闭环
- 证据:
  - 前端仅有 csrf meta 标签: templates/base.html:6
  - 后端未发现 CSRF 校验钩子（无 before_request 校验逻辑）。
- 风险:
  - 一旦后续接入 cookie 会话鉴权，POST/PUT/DELETE 端点将暴露 CSRF 风险。
- 建议:
  - 引入 CSRF 校验中间件并在前端请求头带 token。

### 5) 分页参数缺少统一上限，可能被滥用导致高负载
- 证据:
  - logs size 直接取参: api/logs.py:27
  - warnings size 直接取参: api/warnings.py:72
  - reviews limit 直接取参: api/reviews.py:401
- 风险:
  - 大 page size 请求会放大 DB 与序列化开销。
- 建议:
  - 所有分页参数统一 clamp（如最大 100 或 200）。
  - 返回结果中回显实际生效的 limit。

### 6) 高耗时任务接口缺少频控与并发保护
- 证据:
  - 手动训练接口直接起后台线程: api/model.py:113
  - 补采接口可反复请求触发（虽有运行状态判断但缺乏用户级限流）: api/backfill.py:146
- 风险:
  - 高频触发会造成资源争抢，影响采集稳定性。
- 建议:
  - 增加接口级节流（时间窗 + 调用来源维度）。
  - 增加任务队列状态机（queued/running/finished/failed）并拒绝重复入队。

### 7) 输入校验体系分散，缺少统一 schema
- 证据:
  - 多处 request.get_json 后手工解析: api/routes.py:89, api/holdings.py:629, api/holdings.py:695
- 风险:
  - 类型边界与字段约束不一致，易出现脏数据与异常分支。
- 建议:
  - 引入统一请求校验（Pydantic/Marshmallow）并标准化错误码。

## P2（中期优化）

### 8) 安全响应头可进一步完善
- 现状:
  - 已有 CORS 处理，但缺少 CSP、X-Content-Type-Options、X-Frame-Options、Referrer-Policy 等基线头。
- 建议:
  - 在 after_request 中追加安全头，前端逐步适配 CSP。

### 9) 前端接口可观测性与失败重试策略可增强
- 现状:
  - 页面侧存在轮询与失败提示，但缺少统一 retry/backoff 与请求超时分级。
- 建议:
  - 抽象统一 API 客户端，增加指数退避与错误分类提示。

### 10) 审计追踪信息可结构化
- 现状:
  - 部分关键动作仅写文本日志。
- 建议:
  - 为配置变更、模型导入、补采启动/重试增加结构化审计表（actor、action、target、before/after、request_id）。
